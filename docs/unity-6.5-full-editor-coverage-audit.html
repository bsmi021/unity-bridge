Unity Bridge and the Unity 6.5 Editor API universe

Last updated: 2026-07-09. Baseline: Unity 6000.5.1f1, revision 0d9463e84828, Windows 11.

Verdict: broad control, not full coverage. The bridge has a strong CLI and durable operation layer, but it cannot yet make a defensible claim that every public Unity 6.5 Editor capability is safely reachable and behaviorally proven. The most urgent work is contract and mutation safety, followed by a generated API catalog and a hardened long-tail execution path.

Findings that block a full-coverage claim

P0 - mutation containment and rollback

import-model, asset hashing, and script editing rely on textual prefix checks before or around canonicalization. Traversal and sibling-prefix cases can escape the intended project area; model import can overwrite a destination and then delete it if importer validation fails. Every file mutation needs canonical containment, explicit overwrite behavior, precondition support, and rollback proof.

P0 - inner failures can be outer successes

Several handlers wrap an operation result with success=false inside BridgeResponse.Success. Python trusts the outer response, so a failed mutation can exit zero. This must be fixed centrally and contract-tested across every handler.

P0 - menu validation can execute

Python sends validateOnly, while C# reads validate. The requested validation flag is dropped and the execution branch is selected. Do not use menu --validate-only until this cross-language contract is repaired.

P1 - generic API execution is not a universal fallback

A live 6.5.1f1 probe took about 10.5 seconds, emitted extensive CS1685 duplicate-core-type warnings, and failed a simple LINQ query with ambiguous extension methods. It also serializes results through ToString(), has no durable async contract, and does not automatically record Undo or changed objects.

P1 - eight implemented operations are unreachable through typed CLI paths

Missing wrappers: animation set-curve, add-event, set-properties; terrain set-heights, set-settings; tilemap fill-box, compress-bounds; and asset reserialize.

P1 - parameter, async, and state-restoration defects

  • Screenshot sends camera; C# reads cameraPath.
  • Multi-angle capture leaves Scene View at the last angle instead of restoring it.
  • Build Profile creation returns success before asynchronous initialization completes.
  • Profiler capture accepts frameCount in Python but does not use it in C#.
  • Protocol policy contains stale health-check while the registry uses bridge-status.

Unity 6.5-specific public surfaces to reconcile

The release adds or changes APIs beyond the July 2 backlog. The coverage program must explicitly classify Android window insets, Build Profile helper APIs, asynchronous Project Auditor analysis and auto-fix, Graph Toolkit connect/disconnect, shader variants uploaded to the GPU, VFX compute-shader prewarm, UI Toolkit callback/resource APIs, and platform-specific APIs. Package versions and installed playback modules must be pinned before claiming availability.

Recommended architecture for honest full coverage

  1. Generated API catalog. Snapshot documented types plus exact public assembly metadata for the installed Editor, active build target, defines, and resolved project packages. Key every symbol by canonical assembly-qualified signature and retain package/asmdef provenance.
  2. Coverage registry. Classify each symbol or workflow as typed command, hardened escape hatch, public-but-unwrapped, no public API, external dependency, obsolete, or explicit non-goal. Never infer coverage from a neighboring namespace.
  3. Bidirectional contract generation. One schema should drive Python parameters, camelCase wire payloads, C# models, result/error semantics, timeout/retry policy, and contract tests.
  4. Hardened execution host. Keep dedicated commands for common workflows, but repair the long-tail C# host with curated assembly references, structured values/logs, Undo/change tracking, time limits, cancellation, and durable async completion.
  5. Live fixture project. Compile and behavior-test every mutating/async capability on Unity 6.5. Source-token tests are supporting evidence only.
  6. Continuous drift gate. Diff API snapshots and command contracts on every supported Unity/package upgrade. New or removed symbols must produce an explicit coverage decision.

Definition of full coverage

Full coverage is achieved only when every symbol/workflow in a pinned snapshot has a classification and every claimed reachable path has contract, negative-path, and live Editor proof appropriate to its risk. Python line coverage above 90% remains mandatory, but it is not Editor API coverage.

  • No unexplained public symbols or package assemblies remain in the inventory.
  • No command field, command type, timeout, or result schema differs between CLI, Python, and C#.
  • Every mutation proves containment, overwrite policy, hash conflict behavior, rollback/atomicity, idempotence, and temporary-state restoration.
  • Every async API reaches terminal success or failure through the operation ledger; “requested” is not completion.
  • Every no-public-API or UI-only workflow is explicitly recorded and never presented as covered.
  • Packaged install parity, Unity compile, focused EditMode/PlayMode behavior, full Python suite, and coverage all pass.
