Unity Bridge UTF-8 BOM JSON RCA

Last updated: 2026-06-09

Incident: Unity Bridge CLI commands can fail to parse Unity response and operation JSON files after Unity writes those files with a UTF-8 byte order mark (BOM).

Root cause: the C# atomic write helper wrote bridge JSON through new StreamWriter(stream, Encoding.UTF8), which emits a UTF-8 preamble in this runtime. Python then opened the file with plain encoding="utf-8", preserving the BOM as \ufeff, and json.loads() rejected that initial character.

Current real Unity error observed in the latest decoded console response: Assets\Tests\PlayMode\World\WorldTerrainDetailPainterTests.cs(159,94): error CS0117: 'ZoneLifecycleState' does not contain a definition for 'Loading'.

Impact

  • The Unity-side command can complete and write the response, but the Python CLI may report an infrastructure failure instead of the actual Unity result.
  • Response JSON with a BOM is treated as repeatedly unreadable by DirectBridge._try_read_response(). The command can then time out even though the response file exists.
  • Operation JSON with a BOM can raise outside the response-read retry path through OperationStore.load(), so timeout handling, operation status, and state transitions can crash or mask the real Unity result.
  • Manual decoding with utf-8-sig reveals the underlying Unity payload, confirming the problem is the bridge transport/read layer rather than Unity command execution.

Regression Window

The most likely regression was introduced by the in-flight atomic-write hardening change on branch codex/harden-ledger-atomic-writes.

Before this change, BridgeOperationLedger.WriteAtomic() used File.WriteAllText(tempPath, content). The current working tree replaced that with a custom temp-file writer using StreamWriter(stream, Encoding.UTF8), unique temp paths, retries, and file replacement. The retry/atomicity goal is valid, but the encoding object changed the wire format of bridge JSON files.

Why Python Fails

Python's JSON module documents that an initial BOM is not accepted by the deserializer. A local reproduction in this checkout matched the observed error:

json.loads('\ufeff{"status":"success"}')
# JSONDecodeError: Unexpected UTF-8 BOM (decode using utf-8-sig): line 1 column 1 (char 0)

Decoding the same bytes with utf-8-sig strips the BOM before JSON parsing:

json.loads(raw_bytes.decode("utf-8-sig"))

Applied Fix

  1. Primary writer fix: the C# bridge JSON writer now uses new UTF8Encoding(false, true) and passes that encoding to response, operation snapshot, and JSONL event writes.
  2. Reader hardening: Python response and operation snapshot readers now use utf-8-sig, so existing BOM-prefixed files remain readable after the writer fix.
  3. Nested payload tolerance: DirectBridge._parse_data_json() strips a leading \ufeff before parsing nested dataJson content.
  4. Test coverage: regression tests now cover BOM-prefixed response files, BOM-prefixed operation snapshots, and the C# no-BOM writer contract.

Suggested Verification

  • Unit: response parsing accepts bytes starting with EF BB BF.
  • Unit: OperationStore.load() and list_records() accept BOM-prefixed JSON snapshots.
  • Static C#: the bridge writer uses a no-BOM encoding object for temp-file writes.
  • Runtime: send unity-bridge --project C:\Projects\tms-heim --pretty console read --errors or equivalent and confirm the CLI prints the actual Unity console error instead of the BOM exception.
