Unity Bridge Reload and Domain Operation Resilience Research

Last updated: 2026-06-08

Conclusion: unity-bridge already has the right first layer: it separates heartbeat liveness from command readiness and waits before dispatching while Unity is compiling, updating, reloading assemblies, or changing play mode. The next resilience layer should protect commands that are already accepted or running when Unity reloads the scripting domain. That requires a durable operation ledger, reload generation tracking, atomic terminal responses, and client-side wait semantics that pause or reclassify in-flight waits during editor busy windows instead of reporting generic command failure.

Scope

This research focuses on Unity Editor plugins that communicate with outside clients, using this repository's file-based bridge as the concrete case. The goal is not to prevent Unity from compiling or reloading. The goal is to make client-observed behavior correct while Unity is legitimately busy: either keep waiting, report a retryable busy/reload state, or return a precise interruption result.

Implementation Status

As of 2026-06-08, this repository has the first durable implementation slice in place. The bridge now writes per-command operation snapshots to .claude/unity/operations/<commandId>.json, appends transition events to <commandId>.events.jsonl, exposes domainGeneration in heartbeat health, writes C# responses through an atomic central path, and provides unity-bridge operation status COMMAND_ID plus unity_operation_status for client-side inspection.

The implementation is intentionally conservative for reload recovery: accepted/running operations found after a domain reload are marked interrupted and receive a terminal error response unless a response already exists. Handler-specific resume logic can be added on top of the same ledger for operations that are provably pollable after reload.

What Unity Reloads and Why It Matters

  • Unity reloads code and scene state as part of normal Play mode entry by default. Unity documents this as a tradeoff between iteration time and the degree to which Play mode resembles a fresh application start. Source: Unity Manual: Code and scene reload on entering Play mode.
  • AssetDatabase refresh can compile code-related files, reload the C# domain, import assets, hot reload assets, and restart its refresh loop when import work creates more work. Source: Unity Manual: Refreshing the Asset Database.
  • InitializeOnLoad static constructors run when Unity loads the editor and when scripts are recompiled. Unity warns that asset operations should be avoided there because asset importing may not be complete. Source: Unity Scripting API: InitializeOnLoadAttribute.
  • For initialization that needs assets after a reload, Unity recommends AssetPostprocessor.OnPostprocessAllAssets(..., bool didDomainReload), which is called after importing completes and can safely perform asset database operations. Source: Unity Scripting API: AssetPostprocessor.OnPostprocessAllAssets.
  • AssemblyReloadEvents.beforeAssemblyReload and afterAssemblyReload bracket assembly reloads. Source: Unity Scripting API: AssemblyReloadEvents.
  • Unity exposes EditorApplication.isCompiling, isUpdating, isPlayingOrWillChangePlaymode, and playModeStateChanged as editor-state signals. Sources: isCompiling, isUpdating, isPlayingOrWillChangePlaymode, playModeStateChanged.
  • When domain reload on Play mode entry is disabled, static fields and static event subscriptions persist between Play mode runs, so plugin code must explicitly reset state and avoid duplicate subscriptions. Source: Unity Manual: Enter Play mode with domain reload disabled.

Current Unity Bridge Baseline

Already Good

  • HealthMonitor.wait_for_ready() waits for command readiness, not only heartbeat liveness.
  • DirectBridge.send_command() waits before generating a command ID or writing a command file.
  • HeartbeatGenerator writes busy flags for compile, AssetDatabase update, assembly reload, and play-mode transition.
  • The Python side treats running responses as non-terminal and keeps polling for the final response.

Still Fragile

  • Accepted and running commands are not represented in durable state outside static C# fields.
  • Domain reload can erase callback dictionaries for tests, play-mode transitions, package manager requests, and synchronous lightmap waits.
  • The response timeout starts after dispatch and does not currently distinguish "Unity reloaded while running" from "handler failed to answer."
  • C# response writes are not consistently atomic, even though Python has retry-on-partial-read behavior.

Failure Modes to Design Around

Failure mode Likely cause Client-visible problem Desired behavior
Command sent just before compile/import Unity becomes busy after the Python pre-dispatch gate passed. Normal response timeout fires even though the editor is alive. Pause or reclassify in-flight wait while heartbeat shows busy/reload, bounded by a hard client deadline.
running response never becomes terminal Static callback state is lost during domain reload. Python keeps reading the same running response until command timeout. Recovery scan writes a terminal interrupted_by_reload, recovered, or retryable_busy response.
Duplicate command execution In-memory _processedCommandFiles resets on domain reload while a command file remains. Mutating commands can apply twice or produce conflicting responses. Use an on-disk accepted/executing/terminal ledger and idempotency keys before invoking handlers.
Bridge initializes too early after reload InitializeOnLoad runs before asset importing completes. Asset-dependent handlers fail immediately after reload. Keep boot code file-only and defer asset operations to OnPostprocessAllAssets(... didDomainReload) or delayCall.
Editor remains unresponsive after an asset batch AssetDatabase.StartAssetEditing() is not matched with StopAssetEditing() after an exception. All later bridge commands appear broken. Use try/finally for every asset editing batch. Unity explicitly warns a missing stop call can leave the editor unresponsive. Source: Unity Scripting API: AssetDatabase.StartAssetEditing.

Recommended Resilience Model

Make command lifecycle durable. Treat every command as a small state machine persisted to disk before handler execution. Heartbeat readiness prevents bad dispatch timing; the ledger prevents reloads from stranding or duplicating accepted work.

1. Add a Durable Operation Ledger

Create <project>/.claude/unity/operations/<commandId>.json with a compact schema:

  • commandId, commandType, parametersHash, idempotencyKey
  • state: queued, accepted, running, completed, failed, interrupted, abandoned
  • domainGeneration, acceptedAt, startedAt, lastProgressAt, terminalAt
  • reloadPolicy: safe_to_retry_before_start, pollable_after_reload, must_not_retry_after_start
  • responsePath, commandPath, lastBusyReason, lastError

The bridge should write accepted before invoking a handler and write running before returning a BridgeResponse.Running. Terminal response writing should update the ledger and response file atomically. This makes post-reload recovery an inspection problem instead of a guess.

2. Track Domain Generation

Add a monotonic domainGeneration or reloadGeneration field to the heartbeat. Increment it from editor-side persistent storage after each domain reload, then include it in heartbeat, operation ledger entries, diagnostics, and responses. If a client sees an in-flight command accepted in generation 12 but the heartbeat is now generation 13 and no recovery terminal response exists, it can report a precise reload interruption instead of a generic timeout.

3. Recover After Reload

Add a small BridgeReloadRecovery initializer that runs after reload and scans the operation ledger:

Operation state after reload Recovery action
queued with no acceptedAt Leave it for normal processing, or mark retryable if the command file is gone.
accepted but not running If no side effects started, write retryable interrupted_before_start and leave retry to the client.
running and pollable Rebuild enough state to poll current Unity status and write a recovered terminal response when possible.
running and not pollable Write terminal interrupted_by_domain_reload with retryable based on handler metadata.
terminal response already exists Do nothing except cleanup old ledger files by age.

4. Give Handlers Explicit Retry and Reload Metadata

Not every command can be retried safely. Add handler metadata, either on ICommandHandler or in a registry table:

  • ReadOnly: safe to retry after no terminal response.
  • MutatingIdempotent: retry only with the same idempotencyKey and a ledger check.
  • MutatingNonIdempotent: never auto-retry after accepted.
  • PollableAfterReload: recovery can infer completion from current Unity state.
  • CanTriggerReload: response wait should expect a reload window.

Examples: hierarchy reads are read-only; component edits are mutating and should not auto-retry after acceptance; play mode transitions are pollable after reload by checking current editor state; test runs are usually not recoverable if the test runner callback state is lost.

5. Change Response Waiting Semantics

Keep the existing command response timeout, but split it into two concepts:

  • Operation timeout: how long the Unity operation is allowed to take when the editor is active.
  • Client patience timeout: how long the CLI or MCP call should wait before returning control to the caller.

While a command is in flight, the Python client should keep reading heartbeat readiness. If the heartbeat becomes busy or stale-after-busy, the wait should produce editor_busy or editor_reloading progress rather than immediately consuming the operation timeout. A hard maximum still prevents infinite hangs.

6. Make Response Writes Atomic

Python already writes command files through a temp file and rename. C# should use the same pattern for response files and ledger updates: write <name>.tmp, flush/close, then replace or move into place. Python's partial-read retry is useful, but atomic terminal writes reduce false running loops and malformed-response windows.

7. Normalize Busy Reason Enums

The current C# heartbeat can emit assembly_reload, while Python-derived readiness names use reloading_assemblies. Normalize busy reasons at the C# boundary or in Python parsing. Suggested canonical values: compiling, updating, reloading_assemblies, playmode_transition, domain_recovery.

8. Keep InitializeOnLoad Lightweight

Use InitializeOnLoad for file paths, event registration, heartbeat startup, and recovery scheduling. Do not do asset loading there. For asset-dependent recovery, schedule one pass through EditorApplication.delayCall or AssetPostprocessor.OnPostprocessAllAssets(..., didDomainReload) after import completes. delayCall is a one-shot post-inspector callback. Source: Unity Scripting API: EditorApplication.delayCall.

Anti-patterns to Avoid

Do not use global reload locking as the main fix. Unity documents EditorApplication.LockReloadAssemblies() as a paired lock for short periods when reload is inconvenient. Every lock must be matched with unlock; a leaked lock prevents scripts from unloading. Source: Unity Scripting API: EditorApplication.LockReloadAssemblies.

  • Do not silently retry mutating commands after they were accepted by Unity.
  • Do not depend on static dictionaries as the only record of active async work.
  • Do not leave running response files as the only progress signal.
  • Do not call AssetDatabase batch APIs without try/finally cleanup.
  • Do not treat stale heartbeat as dead editor when recent busy or reload evidence exists.

Implementation Sequence

Phase Work Why first
A Normalize busy reasons, add reload generation to heartbeat, make C# response writes atomic, and expose generation in status / MCP health. Low-risk observability and protocol cleanup that improves diagnosis immediately.
B Add operation ledger, command acceptance states, terminal ledger writes, and stale ledger cleanup. Creates the durable substrate needed for recovery without rewriting every handler at once.
C Migrate long-running handlers to ledger-backed progress: tests, play mode, package manager, build, lightmap, compile/import operations. Targets the handlers most likely to strand clients while waiting.
D Add post-reload recovery scan and handler metadata for retry/recovery behavior. Turns reload interruptions into precise terminal responses.
E Update Python response wait to watch heartbeat, pause/reclassify busy windows, and support operation polling by command ID. Completes the client behavior needed to avoid false failures during reloads.

Validation Plan

  1. Unit-test heartbeat parsing for domainGeneration, normalized busy reasons, stale-after-busy, and busy-to-ready transitions.
  2. Unit-test DirectBridge so no command is written before readiness and in-flight waits classify busy/reload windows separately from command timeout.
  3. Static-test C# heartbeat and response writer contracts when no Unity compile target is available in this repo.
  4. Add Unity integration tests that send a long-running command, trigger AssetDatabase refresh or script compilation, and verify the client gets either a terminal recovered result or a retryable reload interruption.
  5. Add fixture-based recovery tests that leave queued, accepted, running, and terminal ledger files on disk, then simulate reload recovery.
  6. Run python3 -m pytest tests/ and ruff check src/ tests/; when a Unity project is available, run a live compile/test smoke before claiming editor-side completion.

Bottom Line

The remaining client failure risk is no longer primarily "Unity is busy before dispatch." It is "Unity reloads after a command has been accepted and before the final response is written." The robust fix is to persist command lifecycle state, tag each operation with reload/retry semantics, and make the client understand busy/reload progress while waiting.
