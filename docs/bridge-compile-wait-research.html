Unity Bridge Compile Wait Research

Last updated: 2026-06-06

Conclusion: the bridge should distinguish "Editor is alive but busy compiling, importing, or reloading assemblies" from "bridge is down." The first fix should be client-side: wait for an editor-ready state before writing command files, and return a busy/retryable result when readiness times out. The second fix should enrich the heartbeat so the client can also detect AssetDatabase refresh and assembly reload windows.

Observed Failure Mode

The Python client already reads heartbeat health before dispatch. If the heartbeat is stale or missing, it returns a bridge-health error. If the heartbeat says Unity is compiling, it only logs a warning and still writes the command file, then starts the normal command timeout. This means a normal Unity compile, import, or domain reload can look like a failed bridge command.

  • src/unity_bridge/core/bridge.py checks health at dispatch, but only warns on health.is_compiling.
  • src/unity_bridge/core/health.py treats a fresh compiling heartbeat as healthy, which is correct for liveness but incomplete for command readiness.
  • ClaudeCodeBridge/HeartbeatGenerator.cs writes isCompiling, but not isUpdating, assembly reload state, or a ready flag.
  • ClaudeCodeBridge/ClaudeUnityBridge.cs processes command files in EditorApplication.update. That loop cannot be trusted during compile/reload windows.

Unity API Evidence

  • Unity exposes EditorApplication.isCompiling as a read-only signal for whether the editor is compiling scripts. Source: Unity Scripting API: EditorApplication.isCompiling.
  • Unity exposes EditorApplication.isUpdating for AssetDatabase refresh/import state. Source: Unity Scripting API: EditorApplication.isUpdating.
  • Unity's AssetDatabase refresh can compile code, reload the C# domain, and restart the refresh loop when imports generate additional work. Source: Unity Manual: Refreshing the Asset Database.
  • CompilationPipeline.compilationStarted and compilationFinished provide events for compile lifecycle tracking. Source: Unity Scripting API: CompilationPipeline.compilationStarted.
  • AssetDatabase.Refresh imports changed or added assets and can implicitly trigger asset cleanup. Source: Unity Scripting API: AssetDatabase.Refresh.

Recommended Model

Add a separate editor readiness state. "Healthy" should mean the heartbeat proves Unity/bridge liveness. "Ready" should mean the editor is accepting commands: healthy, not compiling, not updating, and not in an assembly reload grace window.

State Meaning Client behavior
Healthy and ready Heartbeat is fresh and no busy flags are set. Write command and wait for response normally.
Healthy but compiling/updating Unity is alive, but command scheduling is unreliable. Wait before writing the command. Do not start the command timeout yet.
Heartbeat stale after busy state Likely assembly reload or editor stall; cannot prove down yet. Classify as editor_busy until a bounded busy grace period expires.
Heartbeat stale without busy context Bridge may be down or editor frozen. Return the current bridge-unhealthy result.

Implementation Shape

Phase 1: Python-only wait gate

  1. Add HealthStatus.ready and HealthStatus.busy_reason. Initially, ready = healthy and not is_compiling.
  2. Add HealthMonitor.wait_for_ready(timeout_seconds, poll_interval). Unlike wait_for_healthy, it should wait through compiling heartbeats.
  3. In DirectBridge.send_command(), before creating the command ID and writing the file, wait for ready when check_health is true.
  4. Use a separate readiness timeout, not the command response timeout. Suggested default: 180 seconds, configurable by UNITY_BRIDGE_EDITOR_READY_TIMEOUT and a future config field.
  5. If readiness expires, return CommandResult(success=False, exit_code=4) with an error such as Unity Editor is busy compiling; command was not sent. This tells an AI client to wait/retry instead of diagnosing a dead bridge.
async def _wait_for_editor_ready(self, timeout: float) -> HealthStatus:
    while within_timeout:
        health = self._health_monitor.check_health()
        if health.healthy and not health.is_compiling and not health.is_updating:
            return health
        await asyncio.sleep(0.5)
    return last_health

Phase 2: Enrich heartbeat

  1. Add isUpdating = EditorApplication.isUpdating to the heartbeat.
  2. Add isPlayingOrWillChangePlaymode if play-mode transitions also cause false timeouts.
  3. Add a small editor-side tracker using CompilationPipeline.compilationStarted, CompilationPipeline.compilationFinished, and assembly reload events to persist the last busy reason and timestamp.
  4. During busy periods, write heartbeat more frequently if EditorApplication.update is still running. Suggested: 1 second while busy, 5 seconds otherwise.

Phase 3: MCP and CLI guidance

  1. Expose ready, busyReason, and editorBusyAgeSeconds through status and unity_health_check.
  2. Add waitForReady to the MCP health tool. Keep waitForHealthy as liveness-only for compatibility.
  3. Return a distinct retryable status for readiness expiration, for example {"success": false, "status": "editor_busy", "retryable": true}.

Important Non-goals

Do not solve this by locking assembly reloads globally. Unity documents EditorApplication.LockReloadAssemblies() as a paired lock for specific inconvenient periods. A leaked lock prevents scripts from unloading and would create a worse class of failure.

  • Do not make every command handler individually sleep while compiling. The client can avoid sending work while the editor cannot process it.
  • Do not reinterpret all stale heartbeats as busy. Only stale heartbeats with recent busy evidence should get a grace period.
  • Do not silently retry mutating commands after a post-send timeout. Retrying mutating commands can duplicate side effects if Unity eventually processed the first command.

Test Plan

  1. Add unit tests for HealthStatus.ready, is_updating, and busy_reason serialization.
  2. Add a wait_for_ready test where the first heartbeat is compiling and the second is ready.
  3. Add a DirectBridge.send_command test proving no command file is written while the editor is busy.
  4. Add a timeout test proving readiness expiration returns a retryable busy result, not a bridge-down result.
  5. Add C# heartbeat tests or static validation for the new heartbeat fields after Phase 2.

Recommended First Patch

Implement Phase 1 first. It is low risk because it changes only client dispatch timing and error classification. It also handles the exact false-negative currently seen by AI clients: a fresh heartbeat saying isCompiling=true should pause dispatch rather than start a command timeout.
