Executive Summary

Unity Bridge can automate nearly all UPM dependency management that Unity exposes through UnityEditor.PackageManager.Client: install, remove, list, search, embed, resolve, clear cache, package a folder into a .tgz, and batch add/remove through one dependency solver pass. The bridge already covers the first-pass operations, but it does not yet expose AddAndRemove, ClearCache, or Pack.

The larger opportunity is package-state intelligence: combine Client.List(), PackageInfo, Packages/manifest.json, and Packages/packages-lock.json to answer "what changed", "what is direct vs transitive", "which packages are outdated", "which packages are deprecated", "which packages came from Git/local/tarball/registry", and "which settings make this dependency graph reproducible".

The current bridge handler stores multiple pending package requests in a dictionary. Unity's Client docs state that methods can only be called in sequence and that overlapping add/remove calls can produce nondeterministic outcomes. Before expanding mutating package automation, add a single-operation queue or reject new package operations while one is pending.

PackageInfo inspection

Unity exposes richer package metadata than the bridge currently serializes. Useful fields include assetPath, category, dependencies, resolvedDependencies, errors, git, isDeprecated, isDirectDependency, packageId, registry, resolvedPath, source, type, version, and versions. Static helpers FindForPackageName, GetAllRegisteredPackages, and IsPackageRegistered can provide fast loaded-package checks that do not need a registry request.

VersionsInfo is the basis for package outdated. It exposes all versions, compatible versions, deprecated versions, latest, latest compatible, and recommended version.

Manifest And Lockfile Automation

Not every Package Manager feature is a public Client method. The project manifest Packages/manifest.json is also a package-manager configuration file. Unity documents these project-level controls:

  • dependencies: direct project dependencies, including registry versions, Git URLs, local folders, and local tarballs.
  • scopedRegistries: custom registry definitions with name, url, and scopes.
  • enableLockFile: deterministic lockfile creation, enabled by default.
  • resolutionStrategy: how Unity selects indirect dependency versions.
  • testables: package test assemblies to load into Unity Test Framework.
  • pinnedPackages: direct dependencies that should stay at the requested version.

The lock file Packages/packages-lock.json records the resolved dependency graph. Unity says a lock file contains the result of dependency resolution, and deleting it forces a refresh of indirect dependency versions. For Unity Bridge, it is valuable read-only evidence for version, dependency depth, source, URL, and resolved edges.

Recommended split: expose manifest mutation through explicit bridge operations, then call Client.Resolve() and verify with Client.List(includeIndirectDependencies: true) plus lockfile inspection. Keep lockfile operations read-only except for an explicit "delete lockfile and resolve" repair command.

Asset Store Boundary

Unity's Package Manager window can manage Asset Store packages, but Asset Store assets split into two shapes: UPM packages and legacy .unitypackage asset packages. UPM-shaped Asset Store packages can be treated like UPM packages once they are available to the project. Legacy asset packages are downloaded and imported through the Package Manager window and then unpacked into Assets/.

I did not find an official public PackageManager.Client API for headless download of acquired Asset Store packages. The practical bridge surface is therefore:

  • Inventory the local Asset Store cache path on Windows: C:\Users\accountName\AppData\Roaming\Unity\Asset Store-5.x.
  • Import an already-downloaded .unitypackage with the existing asset-ext import-package operation.
  • For UPM packages acquired from the Asset Store, use normal package-manager add/list/update flows where Unity exposes a package identifier.
Avoid promising unattended Asset Store download/install unless a supported API or authenticated Unity workflow is verified in-Editor. Local cache import is automatable; acquisition/download from a Unity account is not proven by the public scripting docs.

Suggested Implementation Shape

  • Add a shared C# package request runner that enforces one active Client request and normalizes result serialization by request type.
  • Extend PackageOperationParams with arrays for packagesToAdd and packagesToRemove, fields for packageFolder/targetFolder, and manifest policy fields.
  • Add a C# manifest helper that reads/writes Packages/manifest.json atomically, preserving existing unknown fields.
  • Add a read-only lockfile parser using structured JSON, not string matching.
  • Expose the same surface in CLI first; MCP is deprecated in this repo, so MCP schema updates can be compatibility-only.
  • Tests: Python parameter shaping, C# model serialization expectations, manifest parser fixtures, lockfile parser fixtures, and handler tests for "pending request rejects second request".
